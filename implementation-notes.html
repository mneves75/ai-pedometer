P1Heart-rate display is clobbered on every Motion fallback path

Evidence: AIPedometer/Core/StepTracking/StepTrackingService.swift:493 — applyMotionSnapshot sets todayHeartRateBPM = nil unconditionally. Same at line 134 in the HealthKit-unavailable wheelchair branch. Heart-rate samples live in HealthKit independently of step samples, so wiping HR whenever steps fall back is a real visible regression for users with an Apple Watch.

Reproducer: Apple Watch user opens the app shortly after wakeup when HealthKit step aggregation is still cold (returns 0), causing the Motion fallback to fire. HR display flips from a real BPM value to “No Data”.

Fix plan: Move heart-rate resolution outside the steps/Motion branching. Always attempt resolveLatestHeartRate when HealthKit sync is enabled, and preserve any previously-known sample if the lookup fails. Add a regression test in HealthKitServiceFallbackTests + StepTrackingServiceTests-equivalent that exercises the Motion fallback path with a non-nil HR.

P1GPXRouteParserError has no LocalizedError conformance

Evidence: Shared/Utilities/GPXRouteParser.swift:3 declares enum GPXRouteParserError: Error, Equatable. WorkoutsView.importRoute surfaces error.localizedDescription straight into an alert, which on iOS resolves to the system fallback “The operation couldn’t be completed. (AIPedometer.GPXRouteParserError error 0.)”. Same pitfall for the fileTooLarge guard the cycle-2 audit added on 2026-05-15.

Fix plan: Conform GPXRouteParserError to LocalizedError, register strings in Localizable.xcstrings, and add a regression test for each case ensuring errorDescription is non-nil and localized. Bonus: WorkoutsView should map fileTooLarge → human-friendly limit copy (“File is over 5 MB; try a smaller GPX export”).

P1Foreground refresh skipped when first .active phase arrives before startup completes

Evidence: AIPedometer/App/AppLifecycleCoordinator.swift:46-52 stores lastPhase = scenePhase before the isStartupComplete() guard. On a cold launch, SwiftUI fires .active almost immediately; if startIfNeeded hasn’t finished yet, we silently swallow the only .active event. The next refresh only happens after a background/foreground bounce.

Fix plan: Don’t store lastPhase until after the guards pass (i.e. only when we actually perform the foreground work). Add a startup-coordinator hook that the lifecycle coordinator can listen on, or simpler: record lastPhase only on the success path, and let cold-launch .active trigger a re-entry once startup completes.

P1AppStartupCoordinator.startIfNeeded latches didStart=true before the work that can be cancelled

Evidence: AppStartupCoordinator.swift:41-50 sets didStart = true before awaiting any async work. If the calling Task is cancelled mid-startup (scene phase change, view detach, OS-level cancellation), hasCompletedStartup stays false but didStart is true, and subsequent calls return without retrying. The composition root attempts this from RootView’s .task, which does cancel on view disappearance.

Fix plan: Reset didStart on cancellation (i.e. when Task.isCancelled is true at function exit and hasCompletedStartup is still false), so the next .task tick can pick up startup again.

P1Widget reload throttle doesn’t reset at midnight

Evidence: StepTrackingService.reloadWidgetsIfNeeded:553-573 only refreshes widgets when ≥5 minutes have elapsed AND step delta is ≥200. After midnight rollover, today’s steps reset to 0, which is <200 from yesterday’s final count, so the throttle suppresses the very first refresh the user actually needs. Worst case: a user who crosses midnight at 23:59 with a fresh widget reload sees yesterday’s count on their lock-screen widget for up to 5 minutes.

Fix plan: Always allow a widget reload when the calendar day flips relative to lastWidgetReloadAt. Tests: simulate two consecutive updates across midnight and assert reloadTimelines is invoked the second time.

P1DashboardView Xcode Preview omits PremiumAccessStore

SwiftUI Previews are core to the design feedback loop the user asked us to honor. Same fix pattern as the cycle-3 AboutView preview repair on 2026-05-15.

Fix plan: Inject PremiumAccessStore(forcedPremiumEnabled: true, isTesting: true) in the preview.

P2WatchSyncService.send fans out to WC every CMPedometer tick

Evidence: WatchSyncService.swift:22-66 — every updateSharedData() call encodes JSON, calls updateApplicationContext, optionally sendMessage, and (throttled) transferUserInfo. CMPedometer can fire several updates per second when the user is walking. Application-context churn is mostly free (WC coalesces), but sendMessage when reachable is sent unconditionally.

Design decisions where the spec was ambiguous

D1. Heart-rate semantics on the Motion fallback

The pre-fix code unconditionally zeroed todayHeartRateBPM on every Motion fallback, which silently dropped Apple-Watch BPM whenever HealthKit briefly returned zero steps.

First attempt (cycle 1): only mutate HR when the fallback path could fetch a non-nil value; preserve the previously displayed BPM otherwise. Initial regression test (motionFallbackNeverOverwritesHeartRateWithNil) encoded that behavior.

Refined choice (cycle 2): always overwrite HR whenever sync is enabled, including with nil. Rationale: Apple’s own Watch / Fitness UX surfaces HR only when a fresh sample exists. The earlier choice could pin a stale BPM forever if the watch died and stopped emitting samples. The new behavior matches platform expectations and avoids the “sticky last-known value” class of bug. The original cycle-1 test was replaced by motionFallbackClearsHeartRateWhenHealthKitReturnsNil.

Encoded via: a small HeartRateRefreshAttempt tri-state (attempted: false / true (value: Double?)) so applyMotionSnapshot can distinguish “we didn’t try” from “we tried and got nil”.

D2. Widget reload throttle should always burst at midnight

The spec didn’t mention midnight specifically. I chose to make the throttle release on calendar day change rather than introduce a separate signal. The downside is a small extra reload on each day rollover, which is exactly the point: home-screen widgets show yesterday’s count for up to five minutes otherwise.

D3. AppLifecycleCoordinator: lastPhase commits only on success

I considered an alternative — exposing a startupDidComplete hook the lifecycle coordinator listens for — but rejected it because it widened the API surface for a single-line race. The chosen fix (delay lastPhase commit until after guards pass for .active while leaving non-active transitions instant) reads as a literal “only remember what we actually did”. Cheaper, narrower, equally correct.

D4. AppStartupCoordinator: reset didStart on cancellation

The narrower alternative was a one-shot retry wrapping the call site. I picked the in-class reset because the coordinator is the only owner of didStart and the cancellation is local to its async work. Sub-step idempotency was verified: registerBackgroundTasks already guards re-entry, configureLiveUpdates stops before starting, performInitialSync uses upsert + version-gated cold start.

D5. WatchConnectivity reachable-message throttle

Picked 5 s + 10 step delta as the coalescing window. Rationale: CMPedometer often emits multiple updates per second during a brisk walk; the watch UI doesn’t need a refresh more often than every few seconds. updateApplicationContext still runs every tick (the system itself coalesces that channel), so durability is preserved even when reachable messages are skipped.

Intentional deviations from the existing spec/code

V1. Extracted pure throttle helpers (shouldReloadWidgets, shouldSendReachableMessage)

Previously these decisions were inline branches inside StepTrackingService.reloadWidgetsIfNeeded and WatchSyncService.send. Without a seam, no test could prove the midnight-reset or burst-coalescing behavior. I lifted them to static functions with explicit calendar / interval / delta parameters so the regression tests can exercise every branch on real Date values. The default parameter values match the previous magic numbers (5 minutes, 200 steps, 5 s, 10 steps) so behavior is unchanged outside the bug paths.

V2. GPXRouteParserError conforms to LocalizedError

This is a new public-surface guarantee on the type: each case now has a localized, user-facing description. WorkoutsView already binds error.localizedDescription into the alert, so this is a strict UX upgrade with no API change.

V3. Updated Localizable.xcstrings for the new GPX strings

Per CLAUDE.md the catalog is the source of truth for translations. I added the four new error strings in both en and pt-BR rather than relying on Xcode's automatic extraction (which only fires on a specific build path). JSON validated via jq empty.

V4. SettingsView / AICoachView / DashboardView / HistoryView previews now wire PremiumAccessStore

Each of those views reads the store from the environment but the existing previews omitted it. The fixes use PremiumAccessStore(forcedPremiumEnabled: true, isTesting: true) so the preview operates in the “fully unlocked” state without touching the RevenueCat client. Same pattern that WorkoutsView and the 2026-05-15 AboutView repair already use.

Alternatives considered

T1. Wheelchair distance via HKQuantityTypeIdentifier.distanceWheelchair

Considered: initially deferred because adding a HealthKit read type required entitlement, privacy-manifest, and copy review. Final decision: shipped after that review showed the privacy manifest already covered the Health/Fitness categories and the authorization surface could be tested directly. This note stays here as the decision history, not an open risk.

T2. Stale-HR-forever vs. always-fresh display

See decision D1 above. The earlier choice (preserve last-known HR through transient errors) trades correctness for stability; the new choice (always reflect the latest fetch outcome) trades stability for correctness. Apple’s built-in behavior is the latter, so we match it.

T3. Lifecycle race: state-machine vs. minimal guard

Discussed under D3. A full StartupState enum on the lifecycle coordinator would make the dependency explicit but would force every existing test to wire the new state. The minimal guard removes the bug with one moved line of code and zero new contracts, which is the right shape for an audit fix.

T4. WC coalescing window

Tried 1-second / 50-step versus 5-second / 10-step. Picked the latter: longer interval matches WC’s own coalescing cadence on the system side, and a smaller delta means active walkers still feel the watch react quickly when their pace changes meaningfully (10 steps ≈ 5–8 seconds of brisk walking).

HIGH (correctness) Concurrent refreshTodayData() could make the step count visibly regress

Prior status: the 0.80 cycle justified this as idempotent (“@MainActor serializes synchronous regions”). That justification was wrong. The method is async with many await points (fetchSteps, resolveDistance/Floors/HeartRate, motion fallback); @MainActor serializes only synchronous regions — at each await the actor is released and a second invocation interleaves.

Concrete interleaving: five independent callers fire it with no mutual exclusion — Dashboard .task + pull-to-refresh, AppLifecycleCoordinator foreground, BackgroundTaskService, SettingsSideEffects. App-foreground + a simultaneous background refresh: call A suspends inside fetchSteps; call B runs to completion (sets todaySteps = 9000, seeds the live baseline); call A resumes with its older hkSteps = 8500, overwrites todaySteps = 8500 (visible regression) and re-seeds liveBaseline with stale inputs. Because seedLiveBaseline uses max(healthKitSteps − snapshot.steps, 0), a stale-low read clamps the Apple Watch offset to 0 — the displayed count stays wrong until the next clean refresh.

Fix: serialized the refresh — each call chains after the previous via a stored Task so the body runs to completion atomically, in call order. Deliberately not a coalescing/drop guard (a dropped refresh after a settings change would be a different bug), so every caller still gets a correct, current refresh. StepTrackingService.swift; the public protocol signature is unchanged.
