# Xcode Security Settings

Security build settings decisions for AIPedometer. Maintained by security audits
(`audit-xcode-security-settings` skill); append/update only.

Note: this project is generated by XcodeGen. Build settings live in `project.yml`;
entitlements are rewritten on every `xcodegen generate` by
`Scripts/restore-entitlements.sh` (postGenCommand), so entitlement decisions are
encoded there, never in the `.entitlements` files directly.

## Enabled settings

- `ENABLE_ENHANCED_SECURITY` (2026-06-09): project-level in `project.yml`. Applies to the
  iOS app target (the only supported product type in this project — widgets are an
  app-extension, the watch app is watchOS).
- `ENABLE_POINTER_AUTHENTICATION` (cascaded `YES` from Enhanced Security): arm64e build
  for the iOS app and widget extension. RevenueCat (`purchases-ios-spm`, pinned revision)
  is a source-only SPM package (0 binaryTargets), so no binary-dependency blocker.
  SPM arm64e opt-in lives in `WorkspaceSettings.xcsettings`
  (`iOSPackagesShouldBuildARM64e=YES`), re-asserted by `Scripts/restore-entitlements.sh`
  after every generate.
  - Target override `NO` on `AIPedometerWatch`: watchOS does not support arm64e.
- `ENABLE_USER_SCRIPT_SANDBOXING` (pre-existing).
- `CLANG_ANALYZER_LOCALIZABILITY_NONLOCALIZED` (pre-existing).
- `GCC_TREAT_WARNINGS_AS_ERRORS` / `CLANG_TREAT_WARNINGS_AS_ERRORS` /
  `SWIFT_TREAT_WARNINGS_AS_ERRORS` (pre-existing).

## Deferred

Settings considered but not yet enabled. Revisit them later.

- Hardened-process entitlements on the `AIPedometer` app target
  (`com.apple.security.hardened-process`, `...enhanced-security-version-string = "2"`,
  `...hardened-heap`, `...dyld-ro`, `...platform-restrictions-string = "2"`) —
  STAGED (2026-06-10), not in the default build. Device/archive signing with these keys
  requires the team provisioning profile to include the Enhanced Security capability;
  regenerating that profile needs an Apple ID signed in to Xcode (`xcodebuild
  -allowProvisioningUpdates` reports "No Accounts" on this machine), which is an
  interactive one-time step. Until then the default entitlements stay plain
  (HealthKit + app group); opt in with
  `ENHANCED_SECURITY_ENTITLEMENTS=1 bash Scripts/restore-entitlements.sh` after signing
  in to Xcode and letting it refresh the profile. Simulator-only workflows accept the
  hardened keys today; the gate exists so device installs keep working autonomously.
- `com.apple.security.hardened-process.checked-allocations` (hardware memory tagging /
  MTE family): default-OFF sub-option; needs M5-class/A-series MTE hardware and a
  soft-mode rollout decision. Not auto-enabled by audit policy; revisit when the user
  wants an MTE rollout.
- Basic Clang safety warnings catalog (`GCC_WARN_ABOUT_RETURN_TYPE`, etc.): not applied —
  codebase is pure Swift (no C/C++/ObjC sources); these settings are scoped to C-family
  languages. Re-run the audit if C-family code is ever added.
- `ENABLE_C_BOUNDS_SAFETY` / `ENABLE_CPLUSPLUS_BOUNDS_SAFE_BUFFERS`: not applicable —
  no C/C++ code present.
