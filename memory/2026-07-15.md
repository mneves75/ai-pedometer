# 2026-07-15

## Crash do AIPedometer no iMarcus — diagnóstico (21:00-21:40 BRT)

Relato do usuário: "app crashes on imarcus".

### Evidência (crash logs reais puxados do device)

- iMarcus ficou inacessível no início da sessão (última conexão 2026-07-14 11:54); voltou a responder
  via rede (`usbmux` ConnectionType Network) e `idevicecrashreport -n -k` puxou os relatórios.
- Dois crashes hoje: `AIPedometer-2026-07-15-200032.ips` e `...-200034.ips` (2 s de intervalo =
  launch → crash → relaunch → crash).
- Assinatura: `EXC_BREAKPOINT (SIGTRAP)`, bug_type 309, thread 6, queue
  `com.apple.root.user-initiated-qos.cooperative`, ~0,7 s após o launch. App 0.93 (49) Release,
  iOS 27.0 beta (24A5380h).
- Stack: `_assertionFailure` ← closure em
  `Configuration.APIKeyValidationResult.checkForSimulatedStoreAPIKeyInRelease`
  (código do RevenueCat linkado no app).

### Causa raiz (CONFIRMADA, não é bug de código do app nem do iOS 27)

- `Config/Local.xcconfig` contém uma key RevenueCat de Test Store (prefixo test) (Test Store / Simulated Store key).
- O RevenueCat pinado, em build **não-DEBUG**, ao classificar a key como `.simulatedStore`:
  loga erro → **mostra um alerta** → `fatalError` **depois que o alerta é dispensado**
  (`Sources/Purchasing/Configuration.swift`, `checkForSimulatedStoreAPIKeyInRelease`).
- Por isso o launch de 14/07 "sobreviveu" aos checks de 5/20 s: o `fatalError` só dispara quando
  alguém toca no alerta. Hoje às 20:00 o usuário abriu o app, dispensou o alerta duas vezes → 2 crashes.
- O install de 0.93 (49) em 14/07 foi **Release** (explícito); `install-on-device.sh` default é Debug.
  Test key em Debug é suportada por design (simulated store dev).

### Não-reprodução em simulador (evidência de exclusão)

- App 0.93 buildado (Debug, Xcode estável) e instalado em simulador iOS 27.0 (24A5380g, quase o mesmo
  build do device): onboarding, permissões HealthKit (novo passo do iOS 27: "Past 30 Days vs Full
  History"), dashboard, 5 abas — **sem crash** (mesmo PID do launch ao fim). Debug não dispara o check.

### Observações colaterais (não são o crash)

- iOS 27 sim: botão "Start Workout" na aba Workouts fica parcialmente atrás da tab bar flutuante e,
  mesmo visível após scroll, o tap não faz nada quando Motion & Fitness = "Unavailable" (simulador).
  Possível no-op silencioso; verificar UX em device/iOS 27 depois.
- Runtime watchOS 26.5 (23T570) voltou a aparecer em `simctl list runtimes` — a nota de 2026-06-13
  ("runtime watchOS 26.x removido e indisponível") está possivelmente OBSOLETA; revalidar antes de
  citar o pincer de distribuição.

### Correção

1. App-level (reproducer-first): `AppConstants.RevenueCat.resolveConfiguration` ganha
   `allowsTestStoreAPIKeys` (default `#if DEBUG`), e key de Test Store resolve para `nil` quando não
   permitida → premium **fail closed** (postura já existente) em vez de entregar a key ao SDK que
   crasha de propósito. Testes novos em `AppConstantsTests` (rejeição via bundle e via env; aceitação
   em Debug; key `appl_` intocada).
2. Remediação imediata do device: reinstalar **Debug** no iMarcus.
3. Correção definitiva (decisão do usuário, revenue-critical, NÃO autônoma): criar a key Apple real (prefixo appl) no dashboard RevenueCat (app App Store já existe no ASC, ID 6778799265) conforme
   `docs/revenuecat/apple-payments-setup.md`, e usá-la em builds Release/TestFlight.
