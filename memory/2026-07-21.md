## ~01:50 — Pre-flight red: flaky telemetry test root-caused and fixed (test-only)

- `CoachServiceStreamingTests.slowRendererRecordsAfterRenderStale` failed once in the full-suite
  pre-flight (591 tests, 1 issue), passed in isolation. Root cause (mechanism-proven, not guessed):
  the probe released the blocked first render in the stream task's `defer` IMMEDIATELY after
  `continuation.finish()`, racing the response loop's generation bump on the main actor — if the
  release won, the first render applied as "current" instead of stale → `staleDiscardedAfterRender == 0`.
- Fix: `StreamRenderProbe.blockFirstRenderUntilNextRenderStarts()` — the first render now blocks until
  a SECOND render invocation begins, which structurally guarantees the newer generation was scheduled
  (invocation 2 implies gen ≥ 2) → first apply is always stale. Idempotent release + cancel preserved.
- Verification: 10/10 `-test-iterations 10` repetitions green; full streaming suite re-run in flight.
- Lesson (matches MEMORY's flake rule): isolated re-run first, then mechanism analysis before touching.
  This one WAS a real test defect (load-dependent race), not just noise.
