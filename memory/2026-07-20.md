## 21:10 -03 — Root cause + fix (red→green per reproducer discipline)

**Root cause (device-proven):** on iOS 27 beta, `SystemLanguageModel.default.availability` is a MOVING
TARGET (settings re-evaluation + model-asset validation flip it while the app runs — observed
notEnabled → restricted → available within 80s). The app snapshotted it at init and only re-read on
foreground, so the Dashboard banner rendered a stale reason for a whole session. Screenshot's
`.appleIntelligenceNotEnabled` and the probe's `.modelNotReady` were the same bug at different instants.
NOT an Apple Intelligence settings problem; NOT an app mapping bug.

**Fix (working tree, uncommitted):**
1. `FoundationModelsService`: injected `systemAvailability` seam (default = real getter) +
   `withObservationTracking` on `SystemLanguageModel.default.availability` (model is Observable per
   iOS 27 swiftinterface) → `handleSystemAvailabilityChange()` re-publishes via `refreshAvailability()`
   and re-arms. Permanent public `os_log` of the raw availability value (AppLogger redacts ALL metadata
   by design — this device-state enum is the only signal that cracked the bug).
2. `CoachService.runResponse`: lazily `configureSession()` when session is nil but availability is
   available — the chat previously fell back to "not available" after a live flip until next foreground.

**Tests:** `AIPedometerTests/AI/FoundationModelsServiceTests.swift` (new, 6 tests — proved compile-red
before the seam existed), `CoachServiceStreamingTests.sendRebuildsSessionAfterAvailabilityFlip`
(proved runtime-red: builderCallCount 0, fallback message, .sessionNotConfigured → green after fix).
Full unit suite: 591/591 green pre-coach-fix; final full run in flight.
