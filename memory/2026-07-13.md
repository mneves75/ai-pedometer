## BUILD DEADLOCK — DEFINITIVE ROOT CAUSE (supersedes prior theories)
`sample` of the wedged process nailed it: the `clang -v -E -dM -isysroot .../iPhoneSimulator26.5.sdk -x c -c /dev/null` compiler probe that **SWBBuildService** spawns during `CreateBuildDescription` hangs **100% in a `write()` syscall** (`Command::Print → raw_fd_ostream::write_impl → write`), 0% CPU, for minutes — the child's stdout/stderr pipe fills and SWBBuildService never drains it. Classic pipe-buffer deadlock between the build service and its own external-tool probe. The SAME clang run standalone returns in 0.038s exit 0, so the toolchain is fine. Reproduced 4× this session, independent of load (host was ~13 AND ~410, identical wedge point). **Microsoft Defender Endpoint Security ext `epsext` (PID 32014, hooking AUTH_EXEC) is the prime suspect** — ES exec-hook interference on the spawned clang. `CODE_SIGNING_ALLOWED=NO CODE_SIGNING_REQUIRED=NO` got past GatherProvisioningInputs (ruled that out) but still wedges at the clang probe. NOT an AIPedometer code defect. Mitigation requires excluding `~/Library/Developer/Xcode` (+DerivedData) from Defender real-time scanning — admin/MDM, out of session scope. Until then, simulator build/test/install (incl. iMarcus) cannot be verified on this host. Recovery to unwedge: `pkill -9 -f "xcodebuild -scheme AIPedometer"; pkill -9 SWBBuildService`.
